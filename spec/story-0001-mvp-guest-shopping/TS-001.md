### AC-001 — Product catalogue page

#### TS-001-001 — Paginated product list returns products with metadata

**Preconditions:**
- inventory-service is running and healthy
- Database contains 25 seeded products for the test tenant, each with at least one SKU with `stock_level > 0`

**Action:**
- `GET /products` with header `X-Tenant-ID: {tenantId}` and query params `page=1&per_page=20`

**Expected outcome:**
- HTTP 200
- Response body matches `ProductPage` schema: `data` is an array of 20 `Product` objects, `meta.total` is 25, `meta.page` is 1, `meta.per_page` is 20
- Each product in `data` has `id`, `name`, `skus` (non-empty array), and `created_at`
- Each SKU in each product has `id`, `label`, `price_minor`, and `stock_level`

### AC-012 — Successful order placement

#### TS-001-021 — Successful order returns 201 with confirmed order

**Preconditions:**
- order-service is running
- Valid guest session exists
- inventory-service mock: `POST /stock/reserve` returns 200 with `reservation_id` and `expires_at`
- Payment gateway mock: capture succeeds
- inventory-service mock: `POST /stock/deduct` returns 200
- notification-service mock: `POST /notifications` returns 202

**Action:**
- `POST /checkout/guest/orders` with headers `X-Guest-Session-Token: {token}`, `X-Tenant-ID: {tenantId}` and body:
  ```json
  {
    "email": "grace@example.com",
    "shipping_address": { "line1": "123 Main St", "city": "London", "postal_code": "SW1A 1AA", "country_code": "GB" },
    "shipping_method_id": "{validMethodId}",
    "payment_method": { "type": "card", "token": "tok_visa_test" },
    "lines": [{ "sku_id": "{skuId}", "quantity": 2 }]
  }
  ```

**Expected outcome:**
- HTTP 201
- Response matches `Order` schema: `status` is `confirmed`, `reference` is non-empty (matches pattern like `ORD-XXXXXXXX-XXXX`), `lines` array has 1 entry matching submitted SKU with `quantity: 2`
- `total` is a positive integer (minor currency units)
- `shipping_address` matches the submitted address
- `created_at` is a recent timestamp

---

#### TS-001-022 — Saga executes steps in correct order: reserve → capture → deduct

**Preconditions:**
- order-service is running
- inventory-service mock: `POST /stock/reserve` returns 200 with `reservation_id`
- Payment gateway mock: capture succeeds and returns payment intent ID
- inventory-service mock: `POST /stock/deduct` returns 200
- notification-service mock: `POST /notifications` returns 202

**Action:**
- `POST /checkout/guest/orders` with valid request body (same as TS-001-021)

**Expected outcome:**
- inventory-service mock `POST /stock/reserve` was called exactly once with `order_id` (UUID) and `lines` matching the submitted items
- Payment gateway mock was called exactly once after stock reservation succeeded
- inventory-service mock `POST /stock/deduct` was called exactly once with the `reservation_id` from the reserve response, after payment capture succeeded
- notification-service mock `POST /notifications` was called exactly once after order was confirmed
- Call order verified: reserve → capture → deduct → notify (sequential)

# ADR-001: Guest Checkout Uses Server-Side Session with Short-Lived Token

**Status:** Accepted
**Date:** 2026-04-03
**Authors:** Platform Team
**Reviewers:** Engineering Lead

---

## Context

The platform needs to support Guest Checkout — allowing shoppers to place orders without creating an account. This requires a mechanism to:

1. Identify the shopper across the multiple steps of checkout (address → shipping → payment → confirmation).
2. Associate a cart, shipping choice, and payment intent with that shopper.
3. Prevent session fixation attacks and replay of stale sessions.

Two approaches were evaluated.

---

## Options Considered

### Option A — Stateless JWT Token (Client-Side)

All checkout state is encoded in a signed JWT returned to the client. The client sends it back with every request. No server-side session store is needed.

**Pros:**
- No session store to manage or scale.
- Horizontally scalable without sticky sessions.

**Cons:**
- JWTs cannot be invalidated before expiry. An intercepted token remains valid.
- Storing a growing cart + checkout state in a JWT becomes large and unwieldy.
- Any state update (e.g. selected shipping method) requires issuing a new token.

### Option B — Server-Side Session with Opaque Token (Chosen)

A short-lived session record is stored server-side (Redis). The client receives an opaque token to include in subsequent requests. The server validates the token and loads the session.

**Pros:**
- Sessions can be invalidated immediately on order completion or explicit abandon.
- No sensitive data (cart, payment intent) is exposed to the client.
- Token is small and fixed-size regardless of session payload size.

**Cons:**
- Requires a distributed session store (Redis).
- Adds a dependency on the session store's availability.

---

## Decision

**Option B — Server-Side Session with Opaque Token** is chosen.

The ability to invalidate sessions on completion and the security properties of opaque tokens outweigh the operational cost of a Redis session store, which is already part of our infrastructure for other purposes.

---

## Consequences

- The Order Service will create and own guest sessions (`POST /checkout/guest/sessions`).
- Session tokens are passed via the `X-Guest-Session-Token` request header.
- Sessions expire after **24 hours of inactivity** and are deleted on order placement.
- The Redis session store must have an SLA equivalent to the Order Service itself.
- Future work: if the guest wants to convert to a Customer account post-purchase, the system will link the guest order by email address.

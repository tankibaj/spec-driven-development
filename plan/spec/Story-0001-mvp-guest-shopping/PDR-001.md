# Feature Concept: MVP Guest Shopping Experience

**Feature folder:** `Story-0001-mvp-guest-shopping`
**Status:** Draft
**Author:** Product Team
**Date:** 2026-04-11
**Depends on:** — (none)
**Blocks:** — (none, but all future features build on this foundation)

---

## What

A shopper can browse the product catalog, add items to a cart, complete
checkout as a guest, and receive an order confirmation email — all through
a working storefront. A merchant operations manager can sign into the admin
portal and view a read-only list of orders with order detail drill-down.

This is the minimum viable end-to-end e-commerce flow: the first version of
the platform where all 5 workspaces are running and connected.

---

## Why

**Business problem:** The platform has approved API contracts and backend specs
(Story-0001, Story-0002) but no runnable product. There is no storefront UI for
browsing products, no cart, no working notification delivery, and no admin
visibility into orders. Nothing can be demonstrated, tested end-to-end, or
evaluated by a real merchant.

**Opportunity:**
- Deliver the first end-to-end working product — from browsing a product to
  receiving a confirmation email.
- Validate the full SDD pipeline: contracts → specs → implementation across all
  5 workspaces simultaneously.
- Prove that the service architecture works in practice: `storefront-app` calls
  `order-service` and `inventory-service`, `order-service` calls
  `notification-service`, `admin-app` reads from `order-service`.
- Close the gaps in the existing specs (catalog UI, cart, notification
  implementation, admin portal) that prevent Story-0001 and Story-0002 from
  being truly complete.

**Risk of not building:** The platform remains a collection of API specs with no
working software. No merchant can evaluate it. No shopper can use it. Every
test of the checkout flow requires mocked data. The order confirmation email
specified in Story-0001 AC-008 is a dead letter.

## UX/UI Design

**Needed:** Yes
**Reasoning:** Two apps require new UI that does not yet have design coverage:

1. **Storefront (storefront-app):** The existing WP-001-FE covers the checkout
   flow (4-step form) but the catalog browsing and cart are entirely new.
   Design is needed for: product list page, product detail page, cart (drawer
   or dedicated page), and the navigation connecting catalog → cart → checkout.

2. **Admin portal (admin-app):** This is the first feature for the admin-app.
   Design is needed for: admin login screen, shell layout (sidebar nav, header
   with tenant context), order list (with search + status filter), and order
   detail (line items, shipping address, timeline).

   The admin shell established here becomes the layout foundation for all
   future admin features (inventory management, customer list, reports).

Lightweight wireframes are sufficient for MVP. High-fidelity design can follow
in a subsequent iteration.

## Scope

### Grace's Journey — Storefront (storefront-app + backends)

| Step | Capability | Status |
|------|-----------|--------|
| 1 | Browse product catalog: paginated list, filter by in-stock, product detail | **New** — FE + inventory-service BE |
| 2 | Cart: add items, view cart, update quantities, remove items | **New** — client-side + order-service integration |
| 3 | Guest checkout: shipping info → method → payment → confirm | **New** — FE + order-service BE |
| 4 | Order confirmation email delivered to guest email address | **New** — notification-service BE |

### Priya's View — Admin Portal (admin-app + order-service)

| Step | Capability | Existing Spec? | Gap |
|------|-----------|----------------|-----|
| 5 | Admin sign-in (email + password + MFA per `roles.md`) | None | **New** |
| 6 | Order list: paginated, searchable by order reference or customer email, filterable by status | order-service has `GET /orders` | **New FE** — no admin UI spec |
| 7 | Order detail: line items, shipping address, payment status, lifecycle timeline | order-service has `GET /orders/{orderId}` | **New FE** — no admin UI spec |

> Admin view is **read-only** in this MVP. Cancel and refund actions are
> out of scope (no new `order-service` endpoints required).

### Backend Services Built in This MVP

| Service | What's Built | Notes |
|---------|-------------|-------|
| `inventory-service` | Full catalog + stock API: create product, list, get, stock query, reserve, deduct, release, observability | API contract in `contracts/api/inventory-service.openapi.yaml` |
| `order-service` | Guest sessions, order placement (saga), order list + detail | API contract in `contracts/api/order-service.openapi.yaml` |
| `notification-service` | `POST /notifications`, `GET /notifications/{id}`, email delivery, template rendering, retry | API contract in `contracts/api/notification-service.openapi.yaml` |

### Explicitly Out of Scope

- User accounts, registration, sign-in for shoppers (future feature)
- Authenticated checkout (saved addresses, saved payment methods)
- Admin actions on orders: cancel, refund, add notes (future feature)
- SMS delivery (API accepts the channel; delivery is stubbed — returns `queued`, does not send)
- Product creation or inventory management UI in admin (products created via seed data / API)
- Partial refunds, bulk order actions, analytics dashboards
- Real-time order updates (polling or websockets) in admin
- Multi-currency, multi-language
- Platform-level admin (`platform_ops`/`platform_admin` roles) — merchant roles only

## Assumptions

1. **Cart is client-side.** For the MVP, the cart lives in browser local state
   (Zustand store + localStorage persistence in `storefront-app`). No server-side
   cart API. Cart contents are sent to the order placement API at checkout time.
   Trade-off: cart is not synced across devices, which is acceptable for
   anonymous guest shoppers.

2. **Products are seeded via API.** Merchants create products by calling
   `POST /products` directly (via script or test fixture). No product management
   UI in this MVP.

3. **Notification-service uses a single email provider.** One provider is chosen
   (open question below) and implemented as an adapter. Provider abstraction is
   not required yet.

4. **Templates are file-based.** Email templates are HTML files bundled with the
   `notification-service`. No database-backed template management.

5. **Admin auth is separate from storefront auth.** Per `roles.md`, merchant
   admin users (roles: `merchant_owner`, `merchant_admin`, `merchant_viewer`,
   `merchant_support`) authenticate via email + password with MFA required for
   `merchant_owner` and `merchant_admin`. The admin-app manages its own auth
   flow independent of the storefront guest session mechanism.

6. **This is the primary spec.** There are no prior approved specs to inherit from.
   All capabilities — backend services, storefront UI, admin UI, notifications —
   are specified fresh within this story.

7. **No new order-service endpoints required for the admin view.** The existing
   `GET /orders` and `GET /orders/{orderId}` endpoints in
   `contracts/api/order-service.openapi.yaml` are sufficient for read-only admin.

## Open Questions

- [ ] **Cart persistence:** Client-side only (localStorage + Zustand) or should
  a lightweight server-side cart endpoint be added to `order-service`? Client-side
  is simpler but loses the cart on browser clear or across devices.

- [ ] **Email provider:** SendGrid (simpler SDK, free tier sufficient for MVP) vs.
  AWS SES (cheaper at scale, more AWS setup required)?

- [ ] **Admin auth:** Build auth in-house (JWT + bcrypt in `order-service` or
  a new `auth-service`) vs. an external provider (Auth0, Clerk, Supabase Auth)?
  In-house is simpler for MVP but places the security burden on the team.

- [ ] **Notification failure handling:** If the confirmation email fails all
  retries, should the order record be flagged, and should the admin order detail
  show a "confirmation email not delivered" indicator?

- [ ] **Product seed data:** How many products for the MVP demo environment?
  Should images be real product photos or placeholder thumbnails?

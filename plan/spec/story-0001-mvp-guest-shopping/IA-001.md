## Blast Radius

**Level:** Low
**Services affected:** 5
**Consumers affected:** 0 (greenfield — no existing consumers or running services)
**Reasoning:** This is the first implementation across all 5 workspaces. No production traffic, no existing data, no running consumers. All contract changes are additive — new endpoints, new optional fields, new tables. The risk is internal consistency during parallel development, not breaking existing functionality.

---

## Consumers Affected

| Consumer | Depends On | Impact If Broken |
|----------|-----------|-----------------|
| storefront-app | inventory-service API (catalog), order-service API (checkout) | Cannot display products or complete checkout |
| admin-app | order-service API (auth, orders) | Cannot authenticate admin users or display orders |
| order-service | inventory-service API (stock reserve/deduct/release during saga) | Cannot complete checkout saga |
| order-service | notification-service API (send confirmation email) | Cannot send order confirmation emails |

All consumers are being built in this MVP. No pre-existing consumers are affected.

---

## Recommended Approach

**Approach:** Additive extension
**Reasoning:** All 5 workspaces are greenfield with no existing implementation. Every contract change is additive: new endpoints, new optional fields, new tables. No existing endpoints or response schemas are modified in a breaking way. No API versioning is required. No ADR is needed.

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|-----------|
| Admin auth in order-service becomes architectural debt | Medium | Medium | Document as known tech debt. Auth can be extracted to a dedicated service later. MVP keeps it simple. |
| Client-side cart data lost on browser clear | Low | Low | Acceptable for anonymous guest shoppers per PDR decision. Cross-device sync is out of scope. |
| Price changes between cart add and checkout | Low | Medium | Order-service resolves current prices from inventory-service at checkout. If prices changed, the order total reflects current prices, not cart-displayed prices. Standard e-commerce behavior. |
| `shipping_methods` table and endpoint do not exist yet | High | High | Must be resolved before implementation. Flagged as `[BLOCKS APPROVAL]` open question in FS-001. |
| Notification status mechanism undefined | Medium | Medium | Order-service needs a way to surface notification delivery status to admin-app. Flagged as `[BLOCKS APPROVAL]` open question in FS-001. |
| Product image field absent from contracts | High | High | Catalog UI requires product images. Additive change needed. Flagged as `[BLOCKS APPROVAL]` open question in FS-001. |

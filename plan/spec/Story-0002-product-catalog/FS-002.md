## Goal

Build the `inventory-service` API that manages the product catalogue and stock levels. This service is the authoritative source of product data and stock availability for the entire platform. It must expose both customer-facing catalogue endpoints (browse products) and internal operational endpoints (stock reservation and deduction) called by the order-service during checkout.

**Primary personas:**
- Priya — The Operations Manager (manages products and stock)
- Grace — The Casual Shopper (browses the catalogue, indirectly via storefront)
- Internal: order-service (reserves and deducts stock during checkout)

---

## Background

Before any checkout flow can work, there must be a product catalogue and a stock management system. This feature delivers the foundational `inventory-service`. It must be containerised, expose a health API for orchestration, and be ready to handle concurrent stock reservation requests correctly to prevent overselling.

All monetary values are stored and returned in minor currency units (e.g. cents) to avoid floating-point errors.

## Acceptance Criteria

### AC-001 — Create a product with SKUs

A merchant admin can create a product with one or more SKUs. Each SKU has a price and an initial stock quantity.

**Testable:** `POST /products` with a valid product name and at least one SKU returns HTTP 201. The response includes the created product with an assigned `id` and all SKU `id`s. The initial stock is immediately queryable via `GET /stock/{skuId}`.

---

### AC-002 — List products with pagination

Any caller can retrieve a paginated list of active products. The list can optionally be filtered to show only in-stock items.

**Testable:**
- `GET /products` returns HTTP 200 with a `data` array and a `meta` object containing `total`, `page`, and `per_page`.
- `GET /products?in_stock_only=true` excludes SKUs with zero available stock.
- `GET /products?page=2&per_page=5` returns the correct slice of results.

---

### AC-003 — Get a single product by ID

Any caller can retrieve full details of a product by its UUID, including all its SKUs and their current stock levels.

**Testable:** `GET /products/{productId}` returns HTTP 200 with the product object. `GET /products/{nonExistentId}` returns HTTP 404.

---

### AC-004 — Query stock level for a SKU

Any caller can retrieve the current stock level for a specific SKU. The response must distinguish between total, reserved, and available (= total − reserved) units.

**Testable:** `GET /stock/{skuId}` returns HTTP 200 with `total`, `reserved`, and `available` fields. `available` = `total` − `reserved` at all times.

---

### AC-005 — Reserve stock (internal — called by order-service)

The order-service can place a temporary hold on stock for a set of SKUs. The reservation expires after 15 minutes. Concurrent reservation requests for the same SKU must not allow the total reserved quantity to exceed available stock (no overselling).

**Testable:**
- `POST /stock/reserve` with a valid `order_id` and `lines` returns HTTP 200 with a `reservation_id` and `expires_at` (~15 min from now).
- If any SKU has insufficient available stock, the entire request returns HTTP 409 with a `STOCK_CONFLICT` body listing the conflicting SKUs. No partial reservations are created.
- Two concurrent reservation requests that together exceed available stock: only one succeeds, the other returns HTTP 409.

---

### AC-006 — Deduct reserved stock on order confirmation (internal)

After an order is confirmed and payment captured, the order-service converts the reservation to a permanent stock deduction.

**Testable:** `POST /stock/deduct` with a valid `reservation_id` returns HTTP 200. The SKU's `total` and `reserved` are both decremented by the reserved quantity. A second call with the same `reservation_id` returns HTTP 409 (already deducted — idempotency guard).

---

### AC-007 — Release a reservation on saga compensation (internal)

If payment fails after stock has been reserved, the order-service releases the reservation to restore the stock hold.

**Testable:** `POST /stock/reservations/{reservationId}/release` returns HTTP 204. After release, the SKU's `available` is restored. Releasing a non-existent or already-released reservation returns HTTP 404 (callers treat this as a no-op success).

---

### AC-008 — Service health and observability

The service exposes standard health, readiness, and metrics endpoints for container orchestration and monitoring.

**Testable:**
- `GET /health` returns HTTP 200 with `{"status": "ok"}` at all times the process is running.
- `GET /ready` returns HTTP 200 only when the database connection is available; returns HTTP 503 if the DB is unreachable.
- `GET /metrics` returns a valid Prometheus text format response.

---

## Out of Scope

- Product updates or deletion (future feature).
- SKU price updates (future feature).
- Reservation expiry background job (the 15-minute TTL is noted in the data schema; background expiry is a separate operational feature).
- Multi-currency support (all prices in a single tenant currency for now).
- Image or media management.

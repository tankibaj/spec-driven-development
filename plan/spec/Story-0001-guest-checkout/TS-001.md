### AC-001 — Guest session creation

#### TS-001-001 — Create guest session happy path

**Preconditions:**
- order-service is running.
- Redis is healthy.
- X-Tenant-ID header is set to a valid UUID.

**Action:**
- `POST /checkout/guest/sessions` with empty body.

**Expected outcome:**
- HTTP 201.
- Response body contains:
  - `id` — a valid UUID.
  - `token` — a non-empty opaque string.
  - `expires_at` — an ISO 8601 datetime approximately 24 hours from now.
- Session token is stored in Redis with a TTL of ≤86400 seconds.

---

#### TS-001-002 — Two consecutive sessions produce unique tokens

**Preconditions:**
- order-service is running; Redis is healthy.

**Action:**
- `POST /checkout/guest/sessions` twice in succession with the same tenant.

**Expected outcome:**
- Both calls return HTTP 201.
- The `token` values in the two responses are different strings.

### AC-003 — Shipping method selection

#### TS-001-008 — Valid shipping_method_id → shipping cost reflected in total

**Preconditions:**
- Shipping method `SM-STANDARD` (cost 500 minor units / £5.00) is seeded in the order-service DB.
- Valid guest session token exists.
- inventory-service mock returns a successful reservation.
- Payment gateway mock returns success.

**Action:**
- `POST /checkout/guest/orders` with:
  - `shipping_method_id`: ID of SM-STANDARD.
  - One line with `unit_price_minor` 1000, `quantity` 1 (subtotal = 1000).

**Expected outcome:**
- HTTP 201.
- `order.shipping_cost == 500`.
- `order.total == 1500` (subtotal 1000 + shipping 500 + tax 0).

---

**Expected outcome:**
- HTTP 400.
- No order created.

### AC-004 — Payment entry (no raw card data)

#### TS-001-010 — No raw card data in order response or DB schema

**Preconditions:**
- Valid guest session; inventory and payment mocks return success.

**Action:**
- `POST /checkout/guest/orders` with `payment.token: "tok_test_abc123"` (opaque token, not raw card number).

**Expected outcome:**
- HTTP 201.
- Response body does not contain any field named `card_number`, `pan`, `cvv`, or `expiry`.
- The `order` record in the DB has no `card_number` column (structural schema guarantee).
- `payment_intent_id` in the DB is an external gateway reference string, not card data.

### AC-005 — Order placement happy path

#### TS-001-011 — POST /checkout/guest/orders → 201, confirmed, valid reference

**Preconditions:**
- Valid guest session token.
- inventory-service mock: `POST /stock/reserve` → 200 with `reservation_id`.
- Payment gateway mock: `POST /v1/payment-intents` → 200 with `payment_intent_id`.
- Notification-service mock: `POST /notifications` → 202.

**Action:**
- `POST /checkout/guest/orders` with:
  - Valid email, address, shipping_method_id, payment token, and one line item.

**Expected outcome:**
- HTTP 201.
- `body.status == "confirmed"`.
- `body.reference` matches the regex `^ORD-\d{8}-[A-Z0-9]{4}$`.
- `body.id` is a valid UUID.
- `body.lines` contains the submitted line item.
- `body.total` equals the expected sum.

### AC-006 — Out-of-stock handling

#### TS-001-013 — Inventory returns STOCK_CONFLICT → 409, no charge

**Preconditions:**
- Valid guest session token.
- inventory-service mock: `POST /stock/reserve` → 409 with `{"code": "STOCK_CONFLICT", "conflicts": [{"sku_id": "...", "requested": 1, "available": 0}]}`.
- Payment gateway mock: configured but must NOT be called.

**Action:**
- `POST /checkout/guest/orders` with the conflicting SKU.

**Expected outcome:**
- HTTP 409.
- `body.code == "STOCK_CONFLICT"`.
- `body.conflicts` is a non-empty array listing the out-of-stock SKU.
- Payment gateway mock received zero calls.

---

#### TS-001-014 — No order record created on stock conflict

**Preconditions:**
- Same as TS-001-013.
- Query order count before the request.

**Action:**
- `POST /checkout/guest/orders` with a conflicting SKU.

**Expected outcome:**
- HTTP 409.
- Order count in DB is unchanged (no new order row).

### AC-007 — Idempotent order placement

#### TS-001-015 — Same Idempotency-Key → identical response, one DB row

**Preconditions:**
- Valid guest session token.
- Inventory and payment mocks return success on first call.
- Idempotency-Key header set to a fixed value `"idem-key-001"`.

**Action:**
- `POST /checkout/guest/orders` with `Idempotency-Key: idem-key-001` — first call.
- `POST /checkout/guest/orders` with `Idempotency-Key: idem-key-001` — second call (same payload).

**Expected outcome:**
- Both calls return HTTP 201.
- Response bodies are identical (same `id`, `reference`, `status`).
- Exactly one order row exists in the DB with `idempotency_key = "idem-key-001"`.
- Payment gateway mock called exactly once (not twice).

---

#### TS-001-016 — Different Idempotency-Keys → two distinct orders

**Preconditions:**
- Valid guest session tokens (separate tokens for each call).
- Inventory and payment mocks return success for both calls.

**Action:**
- `POST /checkout/guest/orders` with `Idempotency-Key: key-A`.
- `POST /checkout/guest/orders` with `Idempotency-Key: key-B`.

**Expected outcome:**
- Both calls return HTTP 201.
- The two order `id` values are different.
- Two distinct order records exist in the DB.

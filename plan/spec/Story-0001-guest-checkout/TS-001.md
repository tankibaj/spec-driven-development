## Traceability

Every scenario below traces to exactly one acceptance criterion in FS-001. No orphan scenarios exist.

## AC-001 — Guest session creation

### TS-001-001 · Happy path — session created

**Preconditions:**
- The storefront is accessible.
- No authenticated user session exists.

**Action:**
`POST /checkout/guest/sessions` with an empty request body (no cart ID).

**Expected outcome:**
- HTTP 201 response.
- Response body contains `id` (UUID), `token` (non-empty string), and `expires_at` (ISO 8601 timestamp ~24 h from now).

---

### TS-001-002 · Happy path — session created with existing cart

**Preconditions:**
- An anonymous cart exists with at least one item.

**Action:**
`POST /checkout/guest/sessions` with `{ "cart_id": "<valid-cart-uuid>" }`.

**Expected outcome:**
- HTTP 201 response.
- The returned session is associated with the provided cart.

---

### TS-001-003 · Invalid cart ID

**Preconditions:**
- No cart exists with the provided ID.

**Action:**
`POST /checkout/guest/sessions` with `{ "cart_id": "non-existent-uuid" }`.

**Expected outcome:**
- HTTP 422 response.
- Response body contains a meaningful `message` field.

---

## AC-002 — Collect shipping information

### TS-001-004 · Happy path — valid shipping form submitted

**Preconditions:**
- A valid guest session token exists.
- The checkout is on the shipping information step.

**Action:**
The shopper fills in all required fields (name, valid email, line1, city, postal code, valid country code) and clicks "Continue".

**Expected outcome:**
- No validation errors are displayed.
- The checkout advances to the shipping method selection step.

---

### TS-001-005 · Missing required field — postal code

**Preconditions:**
- A valid guest session token exists.

**Action:**
The shopper submits the shipping form with `postal_code` left blank.

**Expected outcome:**
- The form does not submit.
- A field-level error is displayed adjacent to the postal code input: "Postal code is required."
- The checkout does not advance.

---

### TS-001-006 · Invalid email format

**Preconditions:**
- A valid guest session token exists.

**Action:**
The shopper enters "not-an-email" in the email field and submits the form.

**Expected outcome:**
- The form does not submit.
- A field-level error is shown: "Please enter a valid email address."

---

### TS-001-007 · Optional fields (line 2, state) may be blank

**Preconditions:**
- A valid guest session token exists.

**Action:**
The shopper submits the shipping form with `line2` and `state` left empty but all required fields filled.

**Expected outcome:**
- No validation error for `line2` or `state`.
- Checkout advances to shipping method selection.

## AC-003 — Shipping method selection

### TS-001-008 · At least one shipping method is displayed

**Preconditions:**
- A valid guest session token exists.
- The shopper has provided a valid shipping address.
- At least one shipping method is configured for the tenant.

**Action:**
The shopper reaches the shipping method selection step.

**Expected outcome:**
- At least one shipping method is listed, each showing its name and cost.

---

### TS-001-009 · Selecting a shipping method updates the order total

**Preconditions:**
- Multiple shipping methods are available with different costs.

**Action:**
The shopper selects a shipping method with a cost of €5.99.

**Expected outcome:**
- The displayed order total updates to include €5.99 shipping.
- The total = cart subtotal + €5.99 + applicable tax.

---

## AC-004 — Payment entry

### TS-001-010 · Valid card tokenised — no raw card data logged

**Preconditions:**
- A valid guest session token exists.
- Shipping address and method have been selected.
- The payment gateway SDK widget is rendered.

**Action:**
The shopper enters valid test card details (number, expiry, CVV) in the embedded widget and confirms.

**Expected outcome:**
- The payment gateway SDK returns a `token` string.
- The token (not raw card data) is sent to `POST /checkout/guest/orders`.
- Service logs and the orders database do NOT contain the card number or CVV.

---

### TS-001-011 · Declined card — error displayed

**Preconditions:**
- The payment gateway is configured to decline the test card used.

**Action:**
The shopper enters a card that the gateway declines and submits.

**Expected outcome:**
- A user-facing error message is shown: "Your payment was declined. Please check your card details or try a different card."
- No order record is created.

---

## AC-005 — Order placement — happy path

### TS-001-012 · Successful order placement

**Preconditions:**
- A valid guest session token exists.
- Cart contains 2 items, both in stock (quantity ≥ requested).
- A valid shipping address, shipping method, and payment token are provided.

**Action:**
`POST /checkout/guest/orders` with all required fields.

**Expected outcome:**
- HTTP 201 response.
- Response body: `status: "confirmed"`, `reference` matches pattern `ORD-\d{8}-[A-Z0-9]{4}`.
- An `orders` record exists in the database with `status = 'confirmed'`.
- Stock levels for both SKUs are reduced by the ordered quantities.
- Payment Intent is in `captured` status in the payment gateway.

---

### TS-001-013 · Order total in response matches expected calculation

**Preconditions:**
- Cart: 2× item A at €10.00, 1× item B at €25.00.
- Selected shipping method: €4.99.
- Tax rate: 20%.

**Action:**
`POST /checkout/guest/orders` with the above configuration.

**Expected outcome:**
- `subtotal_minor`: 4500 (€45.00 in cents).
- `shipping_cost_minor`: 499.
- `tax_minor`: 900 (20% of subtotal).
- `total_minor`: 5899 (€58.99).

---

## AC-006 — Out-of-stock handling

### TS-001-014 · Single item out of stock

**Preconditions:**
- Cart contains SKU-A (qty: 2) and SKU-B (qty: 1).
- SKU-A has `available: 1` (insufficient).
- SKU-B has `available: 5`.

**Action:**
`POST /checkout/guest/orders` with the above cart.

**Expected outcome:**
- HTTP 409 response.
- Response body: `code: "STOCK_CONFLICT"`.
- `conflicts` array contains exactly one entry for SKU-A with `requested: 2, available: 1`.
- No Order record is created.
- The shopper's card is NOT charged.

---

### TS-001-015 · All items out of stock

**Preconditions:**
- Cart contains SKU-A (qty: 1) and SKU-B (qty: 1).
- Both SKUs have `available: 0`.

**Action:**
`POST /checkout/guest/orders`.

**Expected outcome:**
- HTTP 409.
- `conflicts` array lists both SKUs.
- No charge, no order.

---

### TS-001-016 · Frontend displays out-of-stock error

**Preconditions:**
- Storefront is configured with the above out-of-stock scenario.

**Action:**
The shopper clicks "Place Order" on the checkout confirmation step.

**Expected outcome:**
- A user-facing message is shown listing the unavailable items by name.
- The "Place Order" button is re-enabled so the shopper can update their cart.

---

## AC-007 — Idempotent order placement

### TS-001-017 · Duplicate request with same idempotency key

**Preconditions:**
- First `POST /checkout/guest/orders` with `Idempotency-Key: abc-123` succeeds (HTTP 201, order created).

**Action:**
A second identical `POST /checkout/guest/orders` with `Idempotency-Key: abc-123`.

**Expected outcome:**
- HTTP 201 response.
- Response body is identical to the first response (same `id`, `reference`, `status`).
- Exactly one Order record exists in the database.
- The payment gateway shows exactly one captured payment.

---

### TS-001-018 · Different idempotency key creates a new order

**Preconditions:**
- First request with `Idempotency-Key: abc-123` succeeded.

**Action:**
A second request with `Idempotency-Key: xyz-456` and the same payload.

**Expected outcome:**
- HTTP 201 response with a **different** order `id` and `reference`.
- Two distinct Order records exist in the database.

---

## AC-008 — Order confirmation email

### TS-001-019 · Confirmation email sent after successful order

**Preconditions:**
- Order placement succeeds (see TS-001-012).

**Action:**
Monitor the Notification Service for outbound calls following the order placement.

**Expected outcome:**
- Within 2 minutes, `POST /notifications` is called with:
  - `channel: "email"`
  - `template_id: "order_confirmation"`
  - `recipient.address`: the guest email provided at checkout
  - `payload` contains `order_reference`, `total`, `shipping_address`, items list, and estimated delivery window.

---

### TS-001-020 · Confirmation email NOT sent if order fails

**Preconditions:**
- Order placement fails due to out-of-stock (see TS-001-014).

**Action:**
Monitor the Notification Service after the failed placement.

**Expected outcome:**
- No call to `POST /notifications` with `template_id: "order_confirmation"` is made.
